GitLab 13.6随自动部署到EC2和使用趋势仪表板发布

GitLab 13.6 released with Auto Deploy to EC2, Usage Trends Dashboard, Full Code Quality Report and much more!

在GitLab，我们致力于提高开发人员的生产率和满意度。 GitLab 13.6具有所有正确的成分，可以帮助您实现所有这些以及更多！ 我们希望您发现这些重要功能以及 此版本中包含 的 60多个新功能和改进 有用。

改进的易用性和自动化以提高效率

为了使您轻松使用带有Amazon Web Services（AWS）的GitLab CI / CD，Auto DevOps现在已扩展为支持AWS，因此您现在可以 使用Auto DevOps 自动部署到Amazon EC2， 而无需使用Kubernetes（例如Auto DevOps之前需要的）。

Docker Hub已对 docker pull 请求 实施速率限制 。 我们已经 减轻了对 我们的SaaS和自托管用户 的影响 ，并提供了共享的方法来 监控 您环境中的 Prometheus的限制 。 我们希望所有用户都能安全地使用其CI / CD管道和Kubernetes集群。 我们正在将“ 依赖代理” 迁移到“所有人”可用的Core。

现在，组所有者在听取社区的反馈以使用更具描述性的默认分支后，可以 为新存储库 （而不是 分支） 配置自定义 默认初始分支名称，从而 具有更大的灵活性 master 。 说到默认值，静态站点编辑器可以 在项目中 使用 默认的合并请求模板 ，从而减少了在提交更新描述后导航到合并请求的需求。

改善可见性，以便更快地制定决策

您无法解决无法找到的问题。 借助13.6，我们对多个仪表板和报告进行了改进，以帮助您更快地制定决策。

通过 合并请求和“ 完整代码质量报告”中 包含 的 代码质量严重性 ，您现在可以快速确定哪些代码质量违规对于合并前的解决至关重要。 感谢代码质量报告 Vicken Simonian 对社区的贡献 ！

我们已经对 Project Security仪表板 进行 了更新， 以包括 最新运行管道安全扫描 的 结果， 以及 动态漏洞趋势图， 以帮助您掌握实时和历史漏洞趋势。 我们还在 合并请求中 添加了 模糊测试结果 以及其他安全结果，并通过添加源文件名和行号来帮助您快速找到确切的崩溃位置并进行修复，从而提高了此报告的可读性。

GitLab自我管理的管理员 现在 可以 查看其组织 在过去12个月中流行功能（例如用户，项目，组，问题和管道） 的使用趋势 。

改进了可扩展性，实现了无缝工作流程

我们的目标是 与 您可能在环境中使用的 其他流行工具很好地配合 使用，即使您仅使用GitLab的一部分，也可以获得无缝的体验。 有了13.6，为了使VS Code内的访问和协作变得容易，我们改进了 VS Code 的 扩展， 以 直接从VS Code 插入片段 ，查看合并请求和问题并对其进行评论，而无需切换到GitLab界面。

除实例和项目级别外，现在还可以在组级别配置 GitLab 集成 -帮助组所有者轻松管理集成。

和更多

为了使您的 每个项目的存储空间 超出 10GB的限制 ，我们最近引入了一个插件， 用于为您的组或个人名称空间 购买更多的存储 空间。 除了 Dependency Proxy之外 ，我们还将 此版本中的 Tracing 移至Core。

这些只是下面介绍的许多新功能和性能改进中的一些亮点。 如果您想预览下个月版本的内容，请查看我们的“ 即将发布的版本” 页面以及我们的 13.7版本的启动 视频系列，其中产品经理将重点介绍下一版本的主要功能。

Git大文件存储（LFS）是Git处理大文件的解决方案之一。 它使用文本指针替换大文件，同时将文件内容存储在远程服务器（如GitLab）上。 它允许开发人员使用Git对非常大的文件进行版本控制，并保持本地主机和服务器之间的Git操作快速高效。

从GitLab下载包含LFS文件的项目的源归档文件，该文件用于导出指向文件的指针而不是文件本身。 这使归档文件更小，但是归档文件无法保存存储库的完整快照。 拥有不完整的存储库通常意味着需要更多的人工工作来单独添加这些LFS文件，以准确导出存储库。

GitLab 13.6现在将导出LFS文件，而不是指向文件的指针，从而可以无缝导出存储库的内容。

弃用日期： 2021年6月22日

删除非高峰时间模式以进行Docker Machine自动缩放

在GitLab Runner 13.0， 版本＃5069中 ，我们为GitLab Docker Machine执行程序引入了新的计时选项。 在GitLab Runner 14.0中，我们计划删除 非高峰时间模式下 的旧配置选项 。

弃用日期： 2021年6月22日

删除成功和失败以完成构建度量标准转换

在GitLab Runner 13.5中，我们介绍了 failed 并 success 声明了一项工作。 为了支持普罗米修斯规则，我们选择了转换 success/failure 到 finished 度量标准。 在14.0中，我们将删除转换。 有关详细信息， 请参阅 问题＃26900 。

弃用日期： 2021年6月22日

在自定义执行程序中删除从step_script到build_script的翻译

在GitLab亚军13.2翻译 step_script 到 build_script 被添加到自定义执行。 在14.0中，“ build_script”阶段将替换为“ step_script”。 有关详细信息， 请参阅 问题＃26426 。

弃用日期： 2021年6月22日

Sidekiq群集队列选择器配置选项已重命名

GitLab包含大量的后台作业队列。 一些管理员可能希望拥有多个后台作业流程，每个后台流程都运行不同的工作负载。

以前，我们仅支持按名称指定为特定进程处理的队列，或者使用实验性选项允许按属性选择队列。

此选项-以前 experimental_queue_selector -不再实验，已重命名为 queue_selector 。 在GitLab 14.0之前 experimental_queue_selector 将继续工作 。

弃用日期： 2021年5月22日

Web应用程序防火墙（WAF）

GitLab 13.6不推荐使用GitLab的Web应用程序防火墙（WAF）。 由于这是一项重大更改，因此将于2021年5月22日在GitLab 14.0中从产品中删除WAF。 GitLab的WAF具有建筑设计固有的局限性，因此很难满足WAF的传统要求。 通过弃用和删除WAF，GitLab将能够集中精力进一步开发产品中可以为用户提供更多价值的其他领域。 当前依赖GitLab WAF的用户可以继续使用 独立于GitLab 的免费和开源 modsecurity 项目。 有关其他详细信息，请参见 弃用问题 。

弃用日期： 2020年11月22日

清除

与Opsgenie的直接链接将被删除

我们正在通过 HTTP端点 集成与Opsgenie和其他警报工具进行更深入的集成， 以便您可以在GitLab界面中查看警报。 因此，在2021年1月22日发布13.8版之后，将不建议使用警报列表中指向Opsgenie的上一个链接。

搬迁日期： 2021年1月22日

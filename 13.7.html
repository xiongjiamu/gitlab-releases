GitLab 13.7 released with merge request reviewers, automatic rollback in case of failure, quick action to clone issues, GitLab Runner container on OpenShift, and much more!

这是您在13.7中可以期待的：

增强项目管理以实现跨协作

合并请求（MR）对于促进交叉协作至关重要，可以将其与相关问题直接关联，提供一个中心位置，以通过注释进行交流，建议代码更改，执行代码审查等。在此版本中，我们添加了合并请求审阅者，该功能通过使审阅更加容易和更有条理来改善代码审阅过程。现在，您将能够快速找出合并请求中涉及的人，或者请求进行正式审查以向他们发送通知。

工作流中的上下文切换和手动任务阻碍了您在组和项目之间进行有效协作的能力。这意味着您花费更少的时间来开发有价值的功能，而花费更多的时间来管理您的项目，这就是为什么能够快速采取行动克隆问题的能力对您简化敏捷规划和项目管理如此重要的原因。

在项目上进行协作并快速迭代以开发应用程序，这意味着您需要能够快速确定问题的重要性顺序，确定所有阻止者，并使用该信息确定下一步的工作重点。现在，您可以按阻止者对问题进行排序，以快速找出哪些问题阻止了其他问题的进展，以及轻松按问题列表中阻止者的数量进行排序。

改进的发布自动化和部署灵活性

您需要灵活性来控制您如何定期组织，自动化和部署应用程序。可靠且频繁地部署应用程序可以更快地将价值带到客户手中。

为了改善GitLab自动发布的方式，我们添加了在发生故障时自动回滚的功能。此功能会自动将失败的部署恢复为上一次成功的部署，并发送自动通知以提醒您状态。您无需手动进行任何更改，并且可以确信在进行修复时，潜在的问题不会导致停机或加剧。

发生故障时自动回滚非常适合进行一项改进，就是可以在“环境”页面中查看部署状态。现在，您可以轻松找到部署状态并确定需要执行的操作，例如停止或回滚部署。

我们还在Red Hat OpenShift上发布了第一个官方支持的GitLab Runner容器Beta版和我们的Certified Runner Operator，为您提供更多使用GitLab发行方式的灵活性。我们正在努力使它很快就可以普遍使用，因此请继续关注将来的发行版。

更可靠，高效的软件包和依赖项管理

您的工作流程取决于多种编程语言，二进制文件，集成和工件，这些都是开发过程的重要输入或输出。您可以更有效地管理软件包和依赖项，从而减少浪费的开发时间，并且出于效率考虑，我们添加了快速查找和查看通用软件包的选项。

我们还对GitLab的依赖项代理进行了改进，顺便说一下，它已在GitLab 13.6的Core中提供。

现在，您可以避免Docker速率限制，并使用Dependency Proxy来加快管道的速度，以确保在缓存DockerHub上托管的容器映像时对可靠性的信心并提高效率。

社区中许多人都希望看到的另一个改进是，Dependency Proxy现在可以与私有项目一起使用，并解决了那些使私有项目的人无法利用此功能的限制。

最后但并非最不重要的一点是，您将能够将预定义变量与Dependency Proxy一起使用，而不是依赖于自己定义的变量或gitlab.ci-yml文件中的硬编码值。这提供了一种更可扩展且更有效的方式来开始代理和缓存图像。

合并请求的审阅者

核心

要求同事检查您的代码应该是贡献代码的日常工作，但这通常不必要地复杂。要求进行审查之类的简单任务可能会导致混乱。例如，您应该怎么问？一封电邮？评论？聊天消息？没有正式的流程，评论可能会不一致并且难以跟踪。以前，一种选择是将审阅者分配给合并请求，但是即使具有这种形式，作者和审阅者都出现在同一受让人域中，这使得其他团队成员很难知道谁在做什么。

GitLab 13.7引入了针对合并请求的审阅者，允许作者向某人请求审阅。新的“审阅者”字段允许以与受让人相似的方式将用户指定为审阅者。审阅者收到通知，邀请他们审阅合并请求。这提供了一个请求审阅的正式过程，并阐明了合并请求中每个用户的角色。

未来的迭代将包括显示合并请求中最相关的审阅者，以及使审阅者成为中心的简化的合并请求批准流程。您可以按照合并请求审阅者分配史诗中的更多详细信息进行操作。

避免Docker速率限制并加速您的管道

核心
起动机
优质
最终
自由
青铜
银
金

为了更快，更可靠的构建，您可以使用依赖代理来缓存Docker Hub上托管的容器映像。但是，当Docker开始对来自Docker Hub的拉取请求实施速率限制时，您会注意到，即使从缓存中拉出映像，Docker也会将其计入限制。这是因为“依赖代理”仅缓存图像的层（或Blob），而不缓存清单，该清单包含有关如何构建给定图像的信息。由于清单是必需的，因此仍然需要拉取请求。这也意味着，如果Docker Hub不可用，则无法拉出映像。

向前，依赖代理将缓存图像的层和清单。因此，第一次拉动时alpine:latest，该图像将被添加到Dependency Proxy缓存中，并且算作一次对您的速率限制的拉动。下次拉动时alpine:latest，即使Docker Hub不可用，也将从缓存中拉出，并且不会计入您的速率限制。

别忘了，从里程碑13.6开始，Dependency Proxy在Core中可用。因此，请尝试一下，让我们知道您的想法。或者更好的是，考虑对一个未解决的问题做出贡献。

我们将弃用logstash和组合格式，仅使用两个选项文本（用于开发）和JSON统一应用程序和访问日志的格式化程序。

弃用日期： 2021年1月22日

弃用Container Registry日志挂钩

容器注册表当前支持只能用于电子邮件通知的日志记录挂钩。

如今，基于日志条目的警报通常由单独的工具处理。据我们所知，我们的用户都没有依赖此功能，GitLab也未使用它。此功能的实现与基础日志库紧密结合，这是我们在不影响可用功能的情况下切换依赖关系的能力的限制。

为了简化注册表功能和配置，我们将放弃对日志记录挂钩的支持。

弃用日期： 2021年1月22日

弃用Container Registry maxidle和maxactive Redis池设置

我们当前为Redis连接池公开的一些配置设置与基础Redis客户端绑定，并且在替代库中没有等效设置。在我们开始改进Redis集成（例如增加对Sentinel的支持）的工作时，我们决定开始着手尝试以功能更丰富的替代方法来替换当前的Redis客户端依赖项，从而更好地为其提供支持。为此，我们需要替换绑定到当前客户端库的当前Redis池配置设置。

我们打算：

  • 删除redis.pool.maxidle和redis.pool.maxactive设置。
  • 添加redis.pool.size（最大连接数），redis.pool.minidle（最小空闲连接数）和redis.pool.maxlifetime（可以重用连接的最大时间）设置。

弃用日期： 2021年1月22日

弃用Bugsnag的Container Registry支持

Bugsnag是容器注册表支持的错误报告服务之一。据我们所知，没有用户依赖此服务，在GitLab我们使用Sentry。为了简化和整合支持的错误报告服务，我们打算添加对Sentry的支持，并删除对Bugsnag的支持。

弃用日期： 2021年1月22日

弃用对NewRelic的Container Registry支持

NewRelic是容器注册表支持的错误报告服务之一。据我们所知，没有用户依赖此服务，在GitLab我们使用Sentry。为了简化和整合支持的错误报告服务，我们打算添加对Sentry的支持，并删除对NewRelic的支持。

弃用日期： 2021年1月22日

不再支持TLS 1.0和1.1的Container Registry支持

出于安全性考虑，由于GitLab不再支持TLS 1.0和TLS 1.1 。我们将对当前支持1.0（默认），1.1、1.2和1.3的GitLab容器注册表进行同样的操作。且默认为1.0。

我们将弃用对TLS 1.0和TLS 1.1的支持，并在使用它们时显示警告日志消息。这些版本的支持将被删除，并且TLS 1.2将成为默认版本。

弃用日期： 2021年1月22日

在GitLab 14.0中弃用一键式GitLab托管应用程序

在GitLab 13.7中，我们弃用一键安装GitLab托管应用程序。尽管它们使从GitLab部署到Kubernetes的入门非常容易，但是社区的总体反馈是，它们对于实际的Kubernetes应用程序不够灵活或不可定制。相反，我们的未来方向将重点在于通过GitLab CI / CD在Kubernetes上安装应用程序，以便在易用性与扩展自定义之间取得更好的平衡。

我们计划在GitLab 14.0版中完全删除一键式托管应用程序。这不会影响集群中现有托管应用程序的运行方式，但是，您将不再能够通过GitLab UI更新修改那些应用程序。我们建议集群管理员计划通过手动或通过CI / CD重新安装现有的托管应用程序来迁移它们。稍后将在我们的文档中提供迁移说明。

弃用日期： 2020年12月17日

弃用使用Docker注册表API v1的请求

GitLab将于2021年1月22日通过Docker注册表v1 API禁用拉取功能.Docker在2019年6月弃用了该功能，该功能使GitLab团队可以专注于为您提供更多价值并针对当前注册表用例的功能和修补程序。

通过完成以下步骤，GitLab上的v1注册表API的现有用户可以移至v2注册表API：

  1. 将您的Docker Engine更新到17.12或更高版本，以便与v2注册表API兼容。
  2. 如果您在GitLab中具有v1格式的内容，则可以使用较新的Docker客户端（比1.12更新的版本）将其移至v2格式，以重建映像并将其推送到GitLab。

弃用日期： 2021年1月22日

GitLab Runner安装程序忽略skel目录

在GitLab Runner 14.0中，skel默认情况下，安装过程将在创建用户主目录时忽略该目录。有关详细信息，请参阅问题＃4845。

弃用日期： 2021年6月22日

将pwsh设为新注册的Windows Runner的默认外壳

在GitLab Runner 13.2中，PowerShell Core支持已添加到Shell执行程序中。在14.0中，pwsh它将是新注册的Windows运行程序的默认外壳程序。WindowsCMD仍可作为Windows运行程序的外壳程序选项使用。有关详细信息，请参阅问题＃26419。

弃用日期： 2021年6月22日

PostgreSQL 11支持

PostgreSQL 12将是GitLab 14.0中最低要求的版本。它为索引编制，分区和一般性能优势提供了重大改进。

从13.7开始，新安装的GitLab将默认为PostgreSQL 12。要手动升级，请运行gitlab-ctl pg-upgrade。

在使用Patroni进行升级之前，多节点数据库实例将需要从repmgr切换到Patroni。然后可以更新地理辅助数据库并重新同步。

弃用日期： 2021年5月22日

从包中删除/ usr / lib / gitlab-runner符号链接

在GitLab转轮13.3，一个符号链接从加入/user/lib/gitlab-runner/gitlab-runner到/usr/bin/gitlab-runner。在14.0中，我们将删除此符号链接，并将运行器安装在中/usr/bin/gitlab-runner。有关详细信息，请参阅问题＃26651。

弃用日期： 2021年6月22日

删除FF_SHELL_EXECUTOR_USE_LEGACY_PROCESS_KILL功能标志

在GitLab Runner 13.1（版本3376）中，我们介绍了Shell执行器sigterm，然后介绍sigkill了该过程。我们还引入了一个新的功能标志，FF_SHELL_EXECUTOR_USE_LEGACY_PROCESS_KILL因此您可以使用以前的过程终止序列。在GitLab Runner 14.0中，版本＃6413，我们将删除功能标记。

弃用日期： 2021年6月22日

删除FF_USE_GO_CLOUD_WITH_CACHE_ARCHIVER功能标志

在GitLab Runner 14.0中，我们将删除FF_USE_GO_CLOUD_WITH_CACHE_ARCHIVER功能标记。有关详细信息，请参阅问题＃27175。

弃用日期： 2021年6月22日

删除Ubuntu 19.10（Eoan Ermine）软件包

Ubuntu 19.10（Eoan Ermine）已于2020年7月17日星期五到期，在GitLab Runner 14.0中，我们将从软件包分发中删除Ubuntu 19.10（Eoan Ermine）。有关详细信息，请参阅问题＃26036。

弃用日期： 2021年6月22日

删除Docker Machine自动缩放的非高峰时间模式配置

在GitLab Runner 13.0，版本＃5069中，我们为GitLab Docker Machine执行程序引入了新的计时选项。在GitLab Runner 14.0中，我们计划删除非高峰时间模式下的旧配置选项。

弃用日期： 2021年6月22日

删除成功和失败以完成构建度量标准转换

在GitLab Runner 13.5中，我们介绍了failed并success声明了一项工作。为了支持普罗米修斯规则，我们选择了转换success/failure到finished度量标准。在14.0中，我们将删除转换。有关详细信息，请参阅问题＃26900。

弃用日期： 2021年6月22日

在自定义执行程序中删除从step_script到build_script的翻译

在GitLab亚军13.2翻译step_script到build_script被添加到自定义执行。在14.0中，该build_script阶段将替换为step_script。有关详细信息，请参阅问题＃26426。

弃用日期： 2021年6月22日

需要Git 2.29或更高版本

在GitLab 13.7和更高版本中，GitLab需要Git 2.29或更高版本。Omnibus GitLab安装包括Git 2.29。从源安装必须手动升级。

弃用日期： 2020年11月22日

清除

与Opsgenie的直接链接将被删除

我们正在通过HTTP端点集成与Opsgenie和其他警报工具进行更深入的集成，以便您可以在GitLab界面中查看警报。因此，在2021年1月22日发布13.8版之后，将不建议使用警报列表中指向Opsgenie的上一个链接。

搬迁日期： 2021年1月22日

Python 2对依赖性扫描的支持已删除

依赖性扫描不再支持2020年1月1日日落的Python 2（通过设置DS_PYTHON_VERSION: 2）。如果需要Python 2，则需要使用容器版本v2.10.0（例如，registry.gitlab.com / gitlab-org / security-products / analyzers / retire.js：2.10.0）或更早的版本。

搬迁日期： 2020年12月15日

删除对Python 2的SAST支持

自2020年1月1日起，Python 2便已终止生命周期（EoL）。作为维护和更新基础SAST分析器的一部分，我们更新了Bandit，它放弃了对Python 2规则的支持。如果仍然需要支持Python 2应用程序，则可以覆盖GitLab SAST CI模板以固定到支持Python 2的早期版本的Bandit。通过固定至先前的版本，您将不会收到Python SAST分析器的最新更新。以下是您可以放入SAST.gitlab-ci.yml模板中的替代代码段：

include:
  - template: Security/SAST.gitlab-ci.yml
bandit-sast:
  variables:
    SAST_ANALYZER_IMAGE: "$SECURE_ANALYZERS_PREFIX/bandit:2.10.0"

搬迁日期： 2020年12月9日

有关升级到GitLab 13.7的重要说明

  • 在升级到13.7的过程中，GitLab Helm图表部署将短暂中断。这是由于对Webservice部署的管理方式进行了更改，从而导致较旧的部署在创建新的部署之前被销毁。当图表移至此新方法时，只会在此版本中发生。
  • 对于GitLab掌舵图表的用户，已添加了一个新的秘密以支持加密的凭据。如果您禁用了共享秘密图表，则需要时，需要通过GitLab 14.0手动创建此秘密。

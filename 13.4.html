GitLab 13.4 released with Vault for CI variables, Kubernetes Agent, and Security Center... and we’re open sourcing feature flags!

GitLab 13.4随Vault发布，用于CI变量，Kubernetes Agent和Security Center…，我们正在为Starter带来功能标志！

在GitLab，我们始终专注于如何通过您喜欢的平台帮助您的团队降低风险，提高效率并加快交付速度。 本月，我们将带来各种优势，这些优势可将可见性扩展到安全性，降低漏洞，提高效率，使用户体验更好，并帮助您的团队更快地部署。 我们希望您发现这些重要功能以及 此版本中 附带的其他53个新功能 有用。

扩展的安全功能

形式正确，本月的发行版为您的GitLab DevSecOps套件增加了一些功能。 首先， 作为构建和部署过程的一部分， 现在可以将存储在HashiCorp Vault中的机密注入CI / CD作业 中。 接下来，希望将代码部署职责分开的组织可以 通过Reporter访问Deployer角色 来 提升特定用户 。 部署者角色遵循最小特权访问的原则，使他们能够批准合并请求并将代码部署到受保护的环境，而无需访问修改代码本身。 降低风险的另一种方法是使用新的 GitLab Kubernetes代理 。 运营商可以从GitLab部署到其Kubernetes群集，而无需将其群集打开到整个Internet。 我们还将通过 GitLab托管Terraform状态 引入对新Terraform状态文件的自动版本控制， 以支持合规性和调试需求。 最后但并非最不重要的一点是，“实例安全性”仪表板已演变为 具有漏洞报告和设置功能 的 GitLab安全中心 。

更好的用户体验和效率

我们通过 从搜索栏 进行 快速导航 来快速跳转到最近的问题，组，项目，设置和帮助主题，从而 改进了全局搜索功能 。 对于 GitLab页面重定向 ， 我们很高兴 能够重定向站点中的各个页面和目录，这使用户能够更高效地部署页面站点。 对于那些希望获得增强的部署信息的人，此版本使您能够 从环境仪表板管理数百个受支持的项目部署 。 多田 🎉

开源贡献亮点

我们将 在MR diffs中 引入 内联代码覆盖率备注 （由 本月的MVP Fabio Huser 贡献 ！），以便开发人员在进行审查时可以直观地看到Merge Request差异中的代码覆盖率。 知道修改后的代码是否包含在单元测试中有助于加快代码审查速度，并加快合并和部署功能的时间。 我们已将 功能标志移至Starter， 并计划 在13.5中将功能标志移至Core 。

介绍GitLab Kubernetes代理

长期以来，GitLab的Kubernetes集成使无需手动设置即可部署到Kubernetes集群。 许多用户都喜欢易用性，而其他用户则遇到了一些挑战。 当前的集成要求您的集群对Internet开放，供GitLab访问。 对于许多组织来说，这是不可能的，因为他们必须出于安全性，合规性或监管目的而锁定群集访问。 要解决这些限制，用户需要在GitLab之上创建自定义工具，否则他们将无法使用该功能。

今天，我们发布了GitLab Kubernetes代理：一种部署到Kubernetes集群的新方法。 该代理在您的集群内部运行，因此您无需将其打开到Internet。 代理通过从GitLab提取新更改来协调部署，而不是GitLab将更新推送到集群。 无论您使用哪种GitOps方法，GitLab都可以满足您的要求。

请注意，这是该代理的第一个版本。 当前，GitLab Kubernetes代理具有配置驱动的设置，并可以通过代码进行部署管理。 尚不支持某些现有的Kubernetes集成功能，例如Deploy Boards和GitLab Managed Apps。 我们的愿景 是最终实现这些功能，并与代理提供新的，注重安全性和合规性的集成。

前端问题是一种灵活方便的方法，可以在旨在由静态站点生成器解析的数据文件中定义特定于页面的变量。 它通常用于设置页面的标题，布局模板或作者，但是当页面呈现为HTML时可以用于将任何类型的元数据传递给生成器。 每个数据文件的最顶部都包含前端内容，通常将其格式设置为YAML或JSON，并且需要一致且准确的语法。 不熟悉特定语法规则的协作者可能会无意中引入无效的标记，从而导致格式问题甚至构建失败。

静态站点编辑器的WYSIWYG编辑模式已经从编辑器中删除了前端内容，以防止出现这些格式错误。 但是，这样一来，您将无法在不恢复编辑文件原始源的情况下修改存储在前端内容中的值。 在GitLab 13.4中，您可以在一个熟悉的基于表单的界面中访问和编辑每个前沿问题字段的值。 单击“ 设置” 按钮将显示一个面板，该面板显示前件中定义的每个键的表单字段。 使用当前值填充字段，编辑其中的任何一个就像在Web表单中键入一样简单。 以这种方式编辑主题，可以避免语法复杂性，并可以完全控制内容，同时确保最终输出的格式一致。

目前，GitLab支持针对应用程序日志的text / json / logstash日志格式和针对访问日志的text / json / combined。 我们将弃用logstash和组合格式，仅使用两个选项（文本（用于开发）和json）统一应用程序和访问日志的格式化程序。

弃用Container Registry日志挂钩

Container Registry支持 日志挂钩 ，目前仅可用于电子邮件通知。

如今，基于日志条目的警报通常由单独的工具处理。 据我们所知，我们的用户都没有依赖此功能，GitLab也未使用它。 此功能的实现与基础日志库紧密结合，这是我们在不影响可用功能的情况下切换依赖关系的能力的限制。

为了简化注册表功能和配置，我们将放弃对日志记录挂钩的支持。

弃用日期： 2021年1月22日

弃用Container Registry maxidle和maxactive Redis池设置

我们当前为Redis连接池公开的一些配置设置与基础Redis客户端绑定，并且在替代库中没有等效设置。 在我们开始改进Redis集成（例如增加 对Sentinel的支持）的工作时 ，我们决定开始着手用功能更丰富，支持更好的替代方法来替换当前的Redis客户端依赖项。 为此，我们需要替换 绑定到当前客户端库 的当前Redis池 配置设置 。

我们打算删除 redis.pool.maxidle 和 redis.pool.maxactive 设置，并添加 redis.pool.size （最大连接数）， redis.pool.minidle （最小空闲连接数）和 redis.pool.maxlifetime （可以重新使用连接的最大时间）。

弃用日期： 2021年1月22日

在最近对GitLab管理员进行调查后，我们决定不弃用此功能。

本 proxy 部分允许将Container Registry设置为 上游存储库 的 本地镜像 。

在GitLab Runner 13.1， 版本＃3376中 ，我们开始发送 sigterm ，然后再 发送 sigkill 到Shell执行程序中的进程。 我们还引入了一个新的功能标志， FF_SHELL_EXECUTOR_USE_LEGACY_PROCESS_KILL 使您可以使用以前的过程终止方法。 在GitLab Runner 14.0中， 版本＃6413 ，我们将删除功能标记。

弃用日期： 2021年6月22日

删除Ubuntu 19.10（Eoan Ermine）软件包

Ubuntu 19.10（Eoan Ermine）已于2020年7月17日星期五到期，在GitLab Runner 14.0中，我们将从软件包分发中删除Ubuntu 19.10。 有关更多详细信息， 请参阅 问题＃26036 。

弃用日期： 2021年6月22日

删除非高峰时间模式以进行Docker Machine自动缩放

在GitLab Runner 13.0， 版本＃5069中 ，我们为GitLab Docker Machine执行程序引入了新的配置设置计时选项。 在GitLab Runner 14.0中，我们将删除旧的配置选项 非高峰时间模式 。

弃用日期： 2021年6月22日

删除成功和失败构建指标转换

在GitLab Runner 13.5中，我们介绍了 failed 并 success 声明了一项工作。 为了支持Prometheus规则，我们选择了to success 和 failure to finished 作为指标。 在14.0中，我们将删除转换。 有关更多详细信息， 请参阅 问题＃26900 。

弃用日期： 2021年6月22日

在自定义执行程序中删除从step_script到build_script的翻译

在GitLab Runner 13.2中，将 step_script to 的翻译 build_script 添加到了自定义执行器。 在14.0中， build_script 舞台将替换为 step_script 。 有关更多详细信息， 请参阅 问题＃26426 。

弃用日期： 2021年6月22日

有关升级到GitLab 13.4的重要说明

在GitLab 13.0中， 旧版存储已被弃用 ，我们宣布 旧版存储 中的所有项目都将自动升级到GitLab 13.2中的新哈希存储。 此背景迁移被 延迟到GitLab 13.4 。

升级到13.4后，仍在旧存储中的所有项目将通过后台迁移自动迁移。
